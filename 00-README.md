# 도메인 명세 인덱스

각 도메인을 "요구사항(무엇을)"과 "결정해야 할 것(어떻게)"으로 나눠 정리했다.
- **요구사항**: 명확하게 확정 — 개발하며 헤매지 않도록.
- **결정해야 할 것**: 정답을 박지 않음 — 방어하며 스스로 정하는 몫.
- **방어 포인트**: 각 도메인에서 "왜?"에 답할 수 있어야 하는 질문들.

★ = 핵심 도메인 (가장 깊게 파는 곳)

> **모델: D2C.** 브랜드가 자사 제품을 소비자에게 직접 판매한다. 관리자가 상품·재고·배송을 운영한다.

```
01 회원·인증
  02 상품·카테고리
    03 재고 ★
      04 포인트 ★
        05 장바구니
          06 주문 ★
            07 결제 ★
              08 배송

```

앞 도메인 없이 뒤가 성립하지 않도록 의존이 쌓이는 순서.

## 1차 목표 (완주 확률을 높이는 마일스톤)

**1차: 01~07** (회원→상품→재고→포인트→장바구니→주문→결제)
- "주문하면 재고 빠지고 포인트 차감되고 상태 바뀐다"까지 완성.
- 동시성·상태머신·원장·트랜잭션 정합성이라는 핵심 학습이 전부 여기 있음.

**2차: 08~10** (배송→클레임→쿠폰)

## 공통 개발 원칙 (모든 도메인에 적용)

1. **고통 후 리팩터링** — 패턴을 미리 깔지 말고, if-else로 아파본 뒤 도입.
2. **결정은 기록(ADR)** — 왜 이렇게 정했는지 남긴다.
3. **실패 재현** — "해결했다"가 아니라 "깨지는 걸 본 뒤 고친다".
4. **방어 세션** — 각 파일의 방어 포인트에 막힘없이 답할 수 있는지 점검.
