# 주문 (Order) 도메인 ★핵심

> 계층: 핵심 도메인 · 투자: 가장 깊게
> 상태 머신 + 주문 불변성 + 트랜잭션 경계의 무대.

---

## 이 도메인의 책임

구매 계약. 무엇을·얼마에·누가 사기로 했는지 확정하고 불변으로 고정한다.
- 책임: 주문 성립, 주문 항목 관리, 주문 상태 전이
- 넘기는 것: 실제 배송(→배송), 취소·반품(→클레임), 대금(→결제)

---

## 핵심 개념 1 — 주문 불변성 (스냅샷)

주문이 성립되는 순간, 상품·가격·옵션 정보를 **복사해서 주문 항목(OrderItem)에 박는다.**
- 카탈로그를 참조로 들고 있으면 안 됨.
- 나중에 상품 가격이 바뀌어도 이 주문의 금액은 불변.
- "고객이 산 것"과 "현재 카탈로그 상태"는 별개.

## 핵심 개념 2 — 상태 머신 (주문만의 상태)

```
PENDING(결제대기) → PAID(결제완료) → CONFIRMED(구매확정)
       │                  │
       └→ CANCELED(취소)   └→ RETURNED(반품완료)   ※ 전이 가능 조건은 네가 정의
```
- 배송 상태는 여기 없음 (배송 도메인의 별도 상태 머신).
- CONFIRMED는 클레임(반품) 가능 기간의 종료 기준.
- 배송완료 후 N일 경과 시 자동 CONFIRMED (배치).
- 클레임과의 관계: 취소(배송 전)→CANCELED, 반품(배송 후)→RETURNED.
  교환은 주문 상태를 바꾸지 않고 클레임 상태로만 관리할지, 별도 상태를 둘지 결정 필요.
- 부분 클레임(주문항목 일부만) 시 주문 전체 상태를 어떻게 표시할지도 결정 필요.

---

## 요구사항 (무엇을)

### 주문 생성
- 장바구니 선택 항목(또는 즉시 구매)으로 생성
- 배송지 선택
- 쿠폰 적용(선택) → 최종 금액 계산
- 재고 차감 + 결제(포인트 차감)가 함께 성공해야 주문 성립

### 주문 조회
- 구매자: 내 주문 목록·상세, 상태
- 관리자: 전체 주문

### 상태 전이
- 정의한 전이 규칙만 허용 (허용 안 된 전이는 거부)
- 구매확정 (수동)

---

## 핵심 과제 — 트랜잭션 경계

주문 생성 = 재고 차감 + 결제 차감 + 주문 기록. 이 셋의 관계:
- 어디까지가 한 트랜잭션(원자적)인가?
- 하나 실패하면 나머지는? (재고 복원, 포인트 복원)
- 어디서부터 이벤트(비동기)로 넘기나?

이미 정한 원칙: **원자성 필요한 건 동기+단일 트랜잭션, 후속처리는 이벤트.**
→ 이 원칙을 주문 생성 구간에 실제로 그어보는 게 과제.

---

## 범위 밖 (지금 안 함)

- 부분 주문 (한 주문의 일부만 따로 처리) — 클레임에서 부분은 다룸
- 주문서 임시저장
- 예약 주문·정기 주문

---

## 결정해야 할 것 (어떻게 — 네가 방어하며 정한다)

1. 상태 전이표를 직접 작성: 어떤 상태 → 어떤 상태가 가능/불가? 각 근거는?
2. 상태 전이 로직을 어디에 둘까? (엔티티 내부 / 서비스 / State 패턴) — if-else로 먼저 짜고 고통 후 리팩터링.
3. 주문 생성의 트랜잭션 경계: 재고·결제·주문을 한 트랜잭션에? 실패 시 롤백 vs 보상?
   - 함께 정할 것: 재고 차감을 "주문 생성 시점"에 할까 "결제 성공 시점"에 할까? PENDING인데 재고가 빠져도 되나?
4. 스냅샷을 어느 시점에 뜨나? 어디까지 복사하나? (가격만? 상품명도? 옵션도?)
5. CANCELED는 어느 상태에서 가능? PAID 이후 취소면 포인트·재고는?
