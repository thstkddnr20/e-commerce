# 회원 (Member) 도메인

> 계층: 일반 도메인 · 투자: 돌아가게만
> 인증은 세션 기반. 여기서 시간을 오래 쓰지 않는다.

---

## 이 도메인의 책임

누가 시스템을 이용하는가를 관리한다. 가입·로그인·권한 확인·배송지.

---

## 요구사항 (무엇을)

### 회원
- 회원가입: 이메일, 비밀번호, 이름 (권한은 BUYER 고정)
- 로그인: 이메일 + 비밀번호 → 세션 생성, 세션 ID 쿠키 전달
- 로그아웃: 세션 무효화
- 비밀번호는 평문 저장 금지 (해시)
- 이메일 중복 불가

### 권한
- BUYER: 구매·충전·주문·클레임 요청
- ADMIN: 상품·재고·배송·클레임·회원 관리 (가입 플로우 없음, 초기 데이터로 생성)
- ADMIN: 전체 관리 (초기 데이터로 생성, 가입 플로우 없음)

### 배송지
- 한 회원이 여러 배송지 보유
- 기본 배송지 1개 지정
- 배송지: 받는사람, 연락처, 주소, 요청사항

### 인증 흐름
- 로그인 성공 → 서버가 세션 생성·저장(식별자 + 권한) → 세션 ID 쿠키 전달
- 요청 시 쿠키의 세션 ID → 서버가 세션 조회 → 권한 확인
- 인증 실패 → 401, 권한 부족 → 403
- 로그아웃 → 서버가 세션 무효화 (즉시 무효)
- 세션 저장소는 서버 내장 메모리 (단일 인스턴스 전제, 재시작 시 소멸 — 지금은 수용)

---

## 범위 밖 (지금 안 함)

- 소셜 로그인, 이메일 인증, 비밀번호 찾기
- 계정 잠금, 로그인 시도 제한
- 회원 탈퇴/휴면

---

## 결정해야 할 것 (어떻게 — 네가 방어하며 정한다)

1. 권한을 어떻게 표현할까? (enum 단일 role vs 다중 role/권한 조합) 왜?
2. 배송지의 "기본 배송지"는 어떻게 보장할까? (한 개만 기본이도록)
3. 회원 정보 수정 시 어디까지 허용? (이메일 변경 가능 여부와 그 이유)
4. 세션에 권한을 담으면, 권한이 바뀌었을 때 기존 세션은 어떻게 되나? (세션은 서버가 통제하니 무효화 가능 — 이걸 어떻게 다룰까?)

---

## 방어 포인트 (스스로 답할 수 있어야)

- 왜 세션 방식을 택했나? 토큰(JWT) 방식과의 트레이드오프는? (서버 상태 보관 vs 무상태 확장성)
- 세션을 서버 내장 메모리에 두면 어떤 제약이 생기나? (다중 인스턴스·재시작) 언제 Redis로 분리해야 하나?
- 비밀번호 해시에 무엇을 쓰고 왜 그것인가?
- ADMIN을 가입 플로우에서 뺀 이유는?
