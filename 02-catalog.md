# 상품 / 카탈로그 (Catalog) 도메인

> 계층: 지원 도메인 · 투자: 구조를 제대로

---

## 이 도메인의 책임

무엇을 파는가를 관리한다. 상품, 카테고리, 옵션.
재고는 이 도메인이 아니다 (재고 도메인이 소유).

---

## 요구사항 (무엇을)

### 상품
- 관리자가 등록/수정/삭제
- 상품: 이름, 설명, 대표 가격, 카테고리, 판매상태(판매중/중지)
- D2C이므로 판매 주체는 브랜드 하나 (상품 소유자 개념 없음)

### 카테고리
- 계층 구조 (대 > 중 > 소)
- 상품은 최하위 카테고리에 속함
- 하나의 상품은 하나의 카테고리를 가짐

### 옵션 / SKU
- 한 상품에 여러 옵션 조합 (예: 색상 x 사이즈)
- 옵션 조합마다 별도 SKU (재고·가격 차이의 단위)
- 재고는 상품이 아니라 SKU 단위로 매겨짐

### 조회
- 카테고리별 상품 목록
- 상품 검색 (이름 기준)
- 상품 상세 (옵션·가격 포함)

---

## 범위 밖 (지금 안 함)

- 상품 이미지 업로드 (URL 문자열로 대체 가능)
- 상품 리뷰·평점
- 추천·랭킹·개인화
- 검색 엔진(ES 등) — DB 검색으로 충분

---

## 결정해야 할 것 (어떻게 — 네가 방어하며 정한다)

1. 카테고리 계층을 어떤 구조로 저장할까? (인접 리스트 / 경로 열거 / 클로저 테이블) 각 트레이드오프는?
2. 상품과 SKU의 관계를 어떻게 모델링할까? 가격은 상품에? SKU에? 둘 다?
3. 관리자가 상품을 "삭제"하면 실제로 지울까, 상태만 바꿀까? 이미 주문된 상품이라면?
4. 옵션 없는 단순 상품도 SKU를 강제로 만들까, 예외 처리할까?

---

## 다른 도메인과의 관계

- 재고: SKU 단위로 재고 도메인이 수량을 소유. 카탈로그는 "무엇"만, 재고는 "몇 개"만.
- 주문: 주문 시 상품/SKU 정보를 **스냅샷으로 복사**해 감 (카탈로그가 나중에 바뀌어도 주문 불변). 카탈로그를 참조로 들고 있으면 안 됨.

---

## 방어 포인트

- 왜 재고를 카탈로그에서 분리했나? (변경 빈도·동시성 관점)
- 카테고리 구조로 그것을 고른 이유와, 조회/수정 시 트레이드오프는?
- 주문이 상품을 참조가 아니라 스냅샷으로 갖는 이유는?
