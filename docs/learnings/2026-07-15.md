# 계층구조를 표현하기 위한 방법

> 2026-07-15 · 계기: 카탈로그의 카테고리 계층 구조 설계 중

## 한 줄로
카테고리를 대분류, 중분류, 소분류로 깔끔하게 구분하기 위한 방법 탐색 중, 클로저(closure) 테이블이라는 개념에 대해서 발견

## 왜 필요한가
- 인접 리스트는 재귀적으로 하위 카테고리를 탐색하는 방법을 사용하지만, 클로저 테이블은 재귀없이 사용가능하다.
- 깊이 확장에 따라 쿼리가 바뀌는 인접 리스트 방식에 비해 클로저는 그렇지 않다.

## 비교

| 항목 | 인접 리스트 | 클로저 |
|---|---|---|
| 테이블 수 | 1 | 2 |
| 구조 이해 | 매우 쉬움 | 중간 |
| 직계 자식 조회 | `parent_id = X` | `ancestor = X AND depth = 1` |
| 전체 후손 조회 | 깊이만큼 조인 / 재귀 CTE | `ancestor = X` |
| 깊이가 쿼리에 | 박힘 | 안 박힘 |
| 깊이 4단 확장 | 쿼리 전부 수정 | 그대로 |
| 노드 추가 | 행 1개 | 클로저도 갱신 |
| 정합성 | FK | FK |
| 저장 공간 | 최소 | 카테고리 수에 비례 (상품과 무관) |

## 클릭 깊이별 조회

| 클릭한 것 | 인접 리스트 | 클로저 |
|---|---|---|
| 의류 (대) | 조인 2번 | `WHERE ancestor = 1` |
| 상의 (중) | 조인 1번 | `WHERE ancestor = 2` |
| 티셔츠 (소) | 조인 0번 | `WHERE ancestor = 3` |
| 쿼리 통일 | 깊이별 분기 필요 | 하나로 |


## 대가는
계층 구조를 나타내는 별도의 테이블이 더 필요하고, 계층이 복잡해지면 테이블 크기가 굉장히 커질 수 있다.

## 내 프로젝트엔
대, 중, 소로 카테고리를 구분하기로 결정하여 depth가 깊어봤자 0~2까지이므로 클로저 테이블을 사용하겠다.
