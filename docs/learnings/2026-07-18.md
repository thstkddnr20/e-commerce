# JPA를 굳이 사용해야 할까

> 2026-07-18 · 계기: Product, Sku 도메인 설계하다가

## 한 줄로
도메인과 엔티티를 분리하는 상황에서 JPA를 사용하는 것이 맞을지에 대한 생각이 들었다.

## 왜 필요한가
어차피 순수한 도메인을 사용할건데 데이터를 굳이 JPA에 종속시켜서 순수 도메인으로 매핑할 필요가 없다고 생각했다.  
오히려 지연로딩으로 인한 N+1 문제등을 야기하여 발목을 잡을 수도 있다고 생각한다.  
하지만 바로 JPA를 사용하지 않는 것은 너무 과한 선택일지도 모르겠다. 편의성이 매우 떨어지기 때문이다.

## 다른 방법 탐색
데이터를 조회하고 가져와서 순수 도메인으로 매핑하는 다른 방법을 탐색하였다.
- JdbcTemplate - RowMapper와 CRUD를 전부 수작업으로 구현해야 하는 반복 비용 때문에 배제
- MyBatis - SQL을 직접 설계해 통제력이 높고 도메인 중심 설계와 잘 어울리나, 단순 CRUD까지 매퍼를 작성해야 하는 비용이 있어 배제
- Spring Data JDBC - 간단한 CRUD를 Interface 수준에서 제공하고 복잡 쿼리는 @Query로 작성가능하다.

또한 다른 블로그들을 탐색한 결과 JDBC로 읽고 JPA로 쓸 수 있다는 내용을 찾았다 (https://www.reddit.com/r/SpringBoot/comments/162naum/spring_data_jpa_or_spring_data_jdbc/?tl=ko)  
하지만 JPA로 데이터를 쓰는 과정에서 영속성 컨텍스트의 갱신을 위해 SELECT 쿼리가 반드시 먼저 일어난 후, 쿼리가 반영된다. 이 부분이 마음에 들지 않았다.

## 대가는
JPA의 주요 기능인 지연로딩, 더티체킹을 사용하지 못한다.

## 내 프로젝트엔
최종적으로 Spring Data JDBC를 사용해보는 것도 나쁘지 않을 것이라고 판단했다.

## 헷갈렸던 것
JPA 사용 시 JPQL 또는 Native Query를 날리지 않는 이상 영속성 컨텍스트를 거친다는 점에 대해서 다시 기억하는 계기가 되었다.  
그래서 JDBC로 읽고 도메인 객체로 매핑하여 작업한 후, 그 변경 사항을 JPA Entity로 재매핑할 경우 영속성 컨텍스트에 갱신하기 위해 SELECT 쿼리 1번,  
변경을 반영하는 쿼리 1번(또는 1번 이상)으로 오버헤드가 발생할 수 있다.
