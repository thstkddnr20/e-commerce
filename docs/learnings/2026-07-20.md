# Kotlin에서의 엔티티 id 선언 방식

> 2026-07-20 · 계기: Long? 타입의 id를 notnull의 Long 타입을 받는 함수로 넘길때 id!!로 전달하다가

## 코드
```kotlin
@Table(name = "members")
class Member(
    id: Long? = null
) {
    @Id
    @Column("id")
    private val id: Long? = id

    fun id(): Long {
        return requireNotNull(id) {
            "저장되지 않은 Address의 ID에는 접근할 수 없습니다."
        }
    }
}
```

## 왜 필요한가

```kotlin
package github.sangwook.ecommerce.member.domain

import org.springframework.data.annotation.Id
import org.springframework.data.relational.core.mapping.Column
import org.springframework.data.relational.core.mapping.Table

@Table(name = "members")
class Member(
    email: String,
    passwordHash: String,
    name: String,
) {
    @Id
    @Column("id")
    val id: Long = 0

    @Column("email")
    var email: String = email
        private set

    @Column("password_hash")
    var passwordHash: String = passwordHash
        private set

    @Column("name")
    var name: String = name
        private set

}
```
이전에는 위와같이 사용하였다. id에 null을 허용하지 않고 0을 사용하여 이 객체가 아직 데이터베이스에 저장되지 않았음을 나타냈다.  
하지만 코틀린 고수인 지인분과 같이 이야기하면서 생각을 바꿨다. id는 기본적으로 null로 지정하는 것이 안전하다는 것  
관련된 내용에 대해서 찾아보았다.

https://blog.jetbrains.com/idea/2026/01/how-to-avoid-common-pitfalls-with-jpa-and-kotlin/
```text
The important thing here is that all persistence providers treat null as a not-provided primary key value. 
They then use a database sequence or an auto-incremented column to generate a primary key value and set it on the entity object. 
Due to this mechanism, the primary key value is null before the entity gets persisted, and changes during the persist operation.
```

블로그에는 다음과 같이 쓰여있다. PK가 null인 것은 아직 영속되지 않았다는 것을 의미하고 DB가 id를 만들어서 넣어준다.  
즉, 저장 전엔 null, 저장 중에는 할당이다.  

나는 단순히 0이라는 매직넘버로 저장되지 않았음을 표현했지만 블로그에서 말하는 내용을 따르기로 결정했다.  
또한 id 프로퍼티를 private으로 선언하여 외부에서 직접 id를 다루지 못하도록 설정하고 별도의 id() 함수를 만들어 Long?이 아닌 Long 타입을 반환한다.
