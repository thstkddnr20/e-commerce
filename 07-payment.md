# 결제 (Payment) 도메인 ★핵심

> 계층: 핵심 도메인 · 투자: 가장 깊게
> 주문 대금을 치르는 "행위". 수단(포인트)과는 다른 층위.

---

## 이 도메인의 책임

주문 대금을 어떻게 치를 것인가. 대금 확정, 수단 선택, 성공/실패 판정, 취소, 멱등성.
- "주문 ↔ 결제" 협력의 주체.
- 포인트에게 차감을 **지시**하고 결과만 받는다. 잔액의 진실은 포인트가 소유.
- 결제는 "얼마를, 어떤 수단으로, 성공했는가"만 안다.

---

## 왜 포인트와 분리하나 (중요)

- 결제 = 행위/절차. 포인트 = 수단/자산.
- 지금은 수단이 포인트 하나뿐이지만, 분리해두면 나중에 쿠폰 결제·복합 결제 확장 가능.
- 결제 도메인이 없으면 그 확장 지점이 사라지고 개념적으로도 틀림.

---

## 요구사항 (무엇을)

### 결제
- 주문으로부터 결제 요청 (금액, 수단=포인트)
- 포인트 차감 지시 → 성공/실패 결과 수신
- 결제 성공/실패를 주문에 반환
- 결제 기록 남김 (주문, 금액, 수단, 상태, 시각)

### 결제 취소
- 전체 취소 / 부분 취소 (클레임 연계)
- 취소 시 포인트 환불 지시(+)

### 멱등성
- 같은 결제 요청이 중복으로 와도 한 번만 처리
- 네트워크 재시도·더블클릭 방어

---

## 결제 상태 (예시 — 직접 확정)

```
REQUESTED → PAID
         ↘ FAILED
PAID → CANCELED / PARTIAL_CANCELED
```

---

## 범위 밖 (지금 안 함)

- 실제 PG 연동 (포인트가 유일 수단)
- 복합 결제(여러 수단 조합)
- 할부·정기결제
- 결제 수수료

---

## 결정해야 할 것 (어떻게 — 네가 방어하며 정한다)

1. 멱등성을 어떻게 보장할까? (멱등키 / 주문+상태 조건부 처리) 왜?
2. 결제와 포인트 차감을 한 트랜잭션에 묶을까, 나눌까? 나누면 정합성은?
3. 부분 취소 시 금액 계산과 포인트 환불은 어떻게? 여러 번 부분취소되면?
4. 결제 실패와 포인트 부족은 같은 실패인가 다른 실패인가? 주문은 어떻게 구분?

---

## 필수 실습

- [ ] 같은 결제 중복 요청 → 한 번만 처리되는지 테스트 (멱등성)
- [ ] 포인트 부족 시 결제 실패 → 주문 상태·재고가 올바르게 되돌아가는지

## 방어 포인트

- 결제와 포인트를 나눈 이유를 "수단 vs 행위"로 설명할 수 있나?
- 멱등성이 없으면 정확히 어떤 시나리오에서 이중 결제가 나나?
- 결제는 왜 포인트 잔액을 직접 관리하지 않고 지시만 하나?
