# E-Commerce 기획서 (D2C)

> D2C(Direct to Consumer) — 브랜드가 자사 제품을 중간 유통 없이 소비자에게 직접 판매한다.
> 판매 주체는 브랜드 하나이며, 관리자가 상품·재고·배송을 운영한다.

## 3. 도메인 구성

도메인은 "현실에서 무엇을 책임지는가", "무엇이 바뀌면 함께 바뀌는가"를 기준으로 나눈다.

### 핵심 도메인

**주문 (Order)** — 구매 계약
- 무엇을·얼마에·누가 사기로 했는지 확정하고 불변으로 고정
- 주문 성립 시 상품 정보를 스냅샷으로 소유 (이후 상품 가격이 바뀌어도 주문은 불변)
- 책임: 주문 성립, 주문 항목(OrderItem) 관리, 주문 상태 전이
- 넘기는 것: 실제 배송(→배송), 취소·반품·교환(→클레임), 대금 처리(→결제)

**재고 (Inventory)** — 팔 수 있는 수량의 진실
- 차감·복원, 초과판매(oversell) 방지
- SKU 단위로 관리
- 동시성 처리의 핵심 지점

**결제 (Payment)** — 주문 대금을 치르는 행위
- 대금 확정, 결제 수단 선택, 성공/실패 판정, 취소·부분취소, 중복 요청 방어(멱등성)
- 결제 수단은 현재 포인트 하나 (확장 시 쿠폰·복합결제 등 추가 여지)
- 포인트에게 차감을 지시하고 결과만 받는다 — 잔액의 진실은 포인트가 소유

**포인트 (Point)** — 결제에 쓰이는 자산 (원장 방식)
- 구매자는 충전해서 사용. 결제 시 차감되는 수단
- **거래 원장(ledger)**: 잔액 컬럼을 두지 않고 모든 이동을 거래 기록으로 쌓기만 함. 기존 기록은 절대 수정하지 않음
- 거래 유형: 충전(+), 결제 차감(−), 취소·환불(+)
- 현재 잔액 = 해당 지갑의 거래 기록 합산 (스냅샷 없음 — 매번 합산)
- 잔액 부족 검증의 동시성: 재고 oversell과 같은 뿌리, 다른 형태. 락 대상이 "행"이 아니라 "지갑"

### 지원 도메인

**상품/카탈로그 (Catalog)** — 무엇을 파는가
- 상품, 카테고리(계층), 옵션/SKU
- 관리자가 등록·운영

**배송 (Delivery)** — 물건이 이동하는 과정
- 주문과 별개의 상태 머신
- 주문당 배송 1건 (단일 판매 주체)

**클레임 (Claim)** — 계약의 사후 변경
- 취소·반품·교환
- 주문·결제·배송을 참조하는 최고 복잡도 도메인 → 독립

**프로모션/쿠폰 (Promotion)** — 가격을 깎는 정책
- 할인 계산, 적용 우선순위, 선착순 발급

### 일반 도메인

**회원 (Member)** — 인증·인가·권한(BUYER/ADMIN)·배송지
- 인증: 세션 기반 로그인 (서버 내장 세션에 보관, 클라이언트는 세션 ID 쿠키)
- 범위 한정: 소셜 로그인·이메일 인증·계정 잠금 등은 다루지 않음. 회원가입/로그인/로그아웃/권한 체크만

**장바구니 (Cart)** — 주문 직전 임시 상태, Redis 기반, 회원 장바구니

## 5. 인증 흐름 (세션 기반)

서버가 세션을 보관하고, 클라이언트는 세션 ID만 쿠키로 들고 다닌다.

```
회원가입 (BUYER)
로그인 → 서버가 세션 생성·저장 → 세션 ID를 쿠키로 전달
요청 시 쿠키의 세션 ID → 서버가 세션 조회 → 회원·권한 확인
로그아웃 → 서버가 세션 무효화
```

- 세션에 담는 것: 회원 식별자, 권한(role)
- 세션 저장소: 서버 내장 메모리 (WAS 세션). 단일 인스턴스 전제, 재시작 시 세션 소멸 — 지금은 이 제약을 수용
- 인증 실패 시 401, 권한 부족 시 403
- 로그아웃은 서버가 세션을 지우면 즉시 무효화됨 (토큰 방식과 달리 서버가 세션을 통제)
- ADMIN은 가입 플로우 없이 초기 데이터로 생성

## 6. 상태 머신 (서로 독립)

주문·배송·클레임은 각각 별개의 상태 머신을 가진다. 하나의 enum으로 뭉치지 않는다.

### 주문 상태
```
PENDING(결제대기) → PAID(결제완료) → CONFIRMED(구매확정)
                 ↘ CANCELED(취소)    ↘ RETURNED(반품완료)
```
- 배송완료 후 N일 경과 시 자동 CONFIRMED (배치)
- 클레임 연계: 취소(배송 전)→CANCELED, 반품(배송 후)→RETURNED. 교환·부분 클레임의 주문 상태 표기는 결정 사항

### 배송 상태
```
READY(배송준비) → SHIPPING(배송중) → DELIVERED(배송완료)
```

### 클레임 상태
```
REQUESTED(접수) → APPROVED(승인) → RETURNING(회수중)
   → INSPECTING(검수) → COMPLETED(완료)
                     ↘ REJECTED(반려)
```

## 8. 도메인 간 협력 원칙

각 도메인은 자기 상태를 바꾸는 규칙을 스스로 소유한다. 남의 상태를 직접 조작하지 않는다.

- **원자성이 필요한 흐름** (동시에 성공/실패해야 함) → 동기 호출 + 단일 트랜잭션
  - 예: 주문 성립 ↔ 재고 차감 ↔ 결제(포인트 차감)
  - 결제는 포인트에 차감을 지시하고 결과만 받는다 → 주문에게 성공/실패만 반환 → 주문이 자기 상태를 바꾼다
- **최종 일관성이 허용되는 후속처리** → 도메인 이벤트(비동기)
  - 예: 주문완료 → 알림 발송
  - 유실돼도 배치·재처리로 복구 가능한 것만 이벤트로 뺀다

## 9. 핵심 처리 흐름

### 주문 생성 ~ 결제 확정 (원자적 구간)
1. 주문 항목 검증
2. 쿠폰 적용 및 최종 금액 계산
3. 재고 차감
4. 주문 생성 (PENDING)
5. 결제 요청 → 결제가 포인트 차감 지시 → 결과 반환
6. 주문이 결과로 상태 전이 (PAID / 실패 시 재고·포인트 복원 + CANCELED)

> ※ 결정 사항: 재고 차감 시점. 위 순서는 "재고 선점 후 결제"(3→4→5)지만,
> "결제 성공 시점에 차감"으로 바꿀 수도 있다. 전자는 결제 전 재고 선점 →
> 결제 실패 시 복원 필요. 후자는 결제까지 재고가 안 잡혀 동시성 창이 넓어짐.
> PENDING(결제대기)인데 재고가 이미 빠지는 게 맞는지 스스로 방어하고 확정할 것.

### 배송 ~ 구매확정
1. 관리자 송장 등록 → 배송 상태 진행
2. 배송완료
3. 구매확정 (수동, 또는 N일 후 자동 배치)

### 클레임 (취소·반품·교환)
1. 주문/배송 상태로 가능 여부 판단
2. 승인 → 회수 → 검수
3. 결제 취소(전체/부분) → 포인트 환불 기록(+) + 재고 복원

## 10. 난이도 핵심 지점

1. **동시성** — 재고 차감, 쿠폰 선착순 발급
2. **트랜잭션 경계** — 주문 생성 구간의 원자성, 어디까지 묶고 어디서 이벤트로 넘길지
3. **멱등성** — 포인트 충전·결제, 주문 중복 요청 방어
4. **포인트 원장** — 잔액을 저장하지 않고 거래 기록의 합으로 계산 (불변 기록)
5. **잔액 검증 동시성** — 재고 oversell과 같은 뿌리, 지갑 단위 락
6. **상태 머신 분리** — 주문/배송/클레임 3개 독립 상태
7. **주문 불변성** — 상품 스냅샷
8. **자동 구매확정** — 배송완료 기준 배치

---

## 11. 개발 진행 순서

```
회원·인증
  → 상품·카테고리
    → 재고
      → 포인트(원장 + 충전)
        → 장바구니
          → 주문(상태 머신 + 스냅샷)
            → 결제(포인트 차감 + 멱등성)
              → 배송
                → 구매확정
                  → 클레임
                    → 프로모션/쿠폰
```
