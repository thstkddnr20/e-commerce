# 배송 (Delivery) 도메인

> 계층: 지원 도메인 · 투자: 구조를 제대로
> 주문과 별개의 상태 머신.

---

## 이 도메인의 책임

물건이 이동하는 과정. 주문의 "계약"과 분리된 "물류"의 상태.
D2C이므로 주문당 배송 1건.

---

## 핵심 개념 — 주문 ≠ 배송

- 주문은 "계약", 배송은 "물류". 변하는 이유가 다르다.
  - 주문 상태는 구매 계약 규칙에 따라 바뀜
  - 배송 상태는 물류 사정에 따라 바뀜
- 그래서 배송 상태는 주문 상태 enum에 섞지 않는다.

## 상태 머신

```
READY(배송준비) → SHIPPING(배송중) → DELIVERED(배송완료)
```

---

## 요구사항 (무엇을)

### 배송 생성
- 결제 완료된 주문 → 배송 건 생성
- 배송: 주문, 배송지, 송장번호, 상태

### 배송 처리 (관리자)
- 송장번호 등록 → SHIPPING
- 배송완료 처리 → DELIVERED (외부 배송사 연동 없이 수동)

### 조회
- 구매자: 내 주문의 배송 상태
- 관리자: 배송 처리 목록

### 연계
- DELIVERED → 구매확정 대상이 됨 (자동확정 배치의 기준 시점)

## 범위 밖 (지금 안 함)

- 실제 배송사 API 연동 (송장 추적)
- 배송비 계산 정책 (고정 또는 무료로 단순화)
- 다중 배송 (한 주문 → 여러 배송)

---

## 결정해야 할 것 (어떻게 — 네가 방어하며 정한다)

1. 배송 건을 어느 시점에 생성할까? (주문 생성 시 / 결제 완료 후)
2. 배송 상태와 주문 상태는 어떻게 연동? DELIVERED가 되면 주문은 어떻게 되나?
3. 배송 상태 변경 → 구매확정 배치로 이어지는 흐름은 동기인가 이벤트인가?
4. 배송지는 주문이 갖나 배송이 갖나? 둘 다면 중복 아닌가?
